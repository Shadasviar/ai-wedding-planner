# Database Scaffold Implementation Plan

## Overview

Implement the database foundation (F-02) for the Wedding Planner app using Drizzle ORM with SQLite. This replaces the temporary seed-user approach from F-01 (auth-scaffold) with a real database-backed user store, enabling persistent user data and future schema evolution.

## Current State Analysis

**What exists now:**
- Auth.js v5 configured with credentials provider (`auth.ts`)
- `src/lib/db/schema.ts` — placeholder interface + `SEED_USERS` array with bcrypt hashes
- `src/lib/db/` directory exists but contains only `schema.ts`
- No database library installed
- No database connection configured
- `.env.local` contains `AUTH_SECRET` from F-01

**What's missing:**
- Database ORM/library (Drizzle)
- SQLite database file and connection
- Users table schema definition
- Migration system
- Database query layer for auth integration

**Key constraints:**
- Must integrate with existing Auth.js config without breaking current auth flow
- LAN-only deployment (no external database services)
- Low-complexity goal — minimal setup, working fast

## Desired End State

After this plan is complete:
- Drizzle ORM installed and configured with SQLite
- `users` table exists with `id`, `username`, `password_hash`, `created_at` columns
- Migrations run automatically on dev startup
- `auth.ts` queries the database for user authentication (replacing `SEED_USERS` lookup)
- Database file stored at `.data/sqlite.db` (or similar project-local path)
- Fail loudly on startup if database is missing/corrupt

**Verification:**
- `npm run dev` starts successfully and creates/connects to SQLite DB
- Auto-migration creates `users` table on first run
- Login with valid credentials works (same `admin`/`wedding2026` test user)
- Invalid credentials still rejected
- TypeScript compiles without errors

### Key Discoveries:

- Drizzle ORM uses a `.ts` schema file + `.sql` migration files generated by `drizzle-kit`
- SQLite connection string is typically `file:./sqlite.db` or absolute path via `process.env`
- Drizzle has excellent TypeScript inference — schema types flow through to queries
- `drizzle-kit` can auto-generate migrations from schema diffs (`npx drizzle-kit generate`)
- For Next.js, the database connection is typically a singleton exported from `src/lib/db/index.ts`

## What We're NOT Doing

- Password recovery/reset flows (manual reset via direct DB access for 2 users)
- Additional tables beyond `users` (guests, services, etc. come in later slices)
- Database admin UI or GUI tooling (direct SQLite file access or `sqlite3` CLI if needed)
- Unit tests for database queries (manual verification via login UI is sufficient)
- Auth.js adapter pattern (JWT sessions stay in Auth.js; DB is only for credential lookup)
- PostgreSQL or Docker setup (SQLite file is perfect for this LAN-only, 2-user scenario)

## Implementation Approach

Install Drizzle ORM + Drizzle Kit, configure SQLite connection, define the users table schema, set up auto-migrations on dev startup, and wire the database query into `auth.ts` to replace the `SEED_USERS` array lookup. Keep the same bcrypt password comparison pattern from F-01.

## Critical Implementation Details

- **Drizzle Kit for migrations**: Use `npx drizzle-kit generate` to create migration SQL files from schema changes, then run them via `npx drizzle-kit migrate` or auto-run on dev start.
- **Database singleton pattern**: Next.js hot-reload can create multiple DB connections. Export a single `db` instance from `src/lib/db/index.ts` and import it everywhere.
- **SQLite file location**: Store at `.data/sqlite.db` (or `.db/sqlite.db`) — a hidden directory at project root that's easy to `.gitignore` and back up.
- **Bcrypt still required**: Drizzle handles schema/queries, not password hashing. Keep `bcryptjs` for password comparison.

## Phase 3: Define Users Schema

### Overview

Define the users table schema using Drizzle's relational API.

### Changes Required:

#### 1. Create Drizzle schema

**File**: `src/lib/db/schema.ts`

**Intent**: Replace the placeholder interface with a Drizzle table definition for users.

**Contract**: Replace the file content:

```typescript
import { sqliteTable, text, integer } from 'drizzle-orm/sqlite-core'

export const users = sqliteTable('users', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  username: text('username').notNull().unique(),
  passwordHash: text('password_hash').notNull(),
  createdAt: integer('created_at', { mode: 'timestamp' }).notNull().$defaultFn(() => new Date()),
})

// Type exports for use in app code
export type User = typeof users.$inferSelect
export type NewUser = typeof users.$inferInsert
```

**Note**: The schema uses `passwordHash` (camelCase) as the property name but `password_hash` (snake_case) as the column name for SQL consistency.

#### 2. Export helper for auth

**File**: `src/lib/db/schema.ts`

**Intent**: Export a helper function to find a user by username for auth.ts to use.

**Contract**: Add to the file:

```typescript
import { eq } from 'drizzle-orm'
import { db } from '.'

export async function findUserByUsername(username: string) {
  return db.select().from(users).where(eq(users.username, username)).get()
}
```

## Phase 5: Set Up Auto-Migration on Dev Start

### Overview

Configure migrations to run automatically when the dev server starts.

### Changes Required:

#### 1. Create migration runner

**File**: `src/lib/db/migrate.ts`

**Intent**: Create a script that runs pending migrations on startup.

**Contract**: Create the file:

```typescript
import { drizzle } from 'drizzle-orm/better-sqlite3'
import Database from 'better-sqlite3'
import { migrate } from 'drizzle-orm/better-sqlite3/migrator'
import { mkdirSync } from 'fs'
import { dirname, resolve } from 'path'

const dbPath = process.env.DATABASE_PATH ?? '.data/sqlite.db'
mkdirSync(dirname(dbPath), { recursive: true })

const sqlite = new Database(dbPath)
const db = drizzle(sqlite)

export function runMigrations() {
  migrate(db, { migrationsFolder: resolve('./drizzle/migrations') })
}
```

#### 2. Wire into Next.js dev startup

**File**: `next.config.ts`

**Intent**: Run migrations when the dev server starts.

**Contract**: Update the config:

```typescript
import type { NextConfig } from 'next'

const nextConfig: NextConfig = {
  // Run migrations in dev
  experimental: {
    // Option: use a custom server or app-level initialization
  },
}

// In dev mode, run migrations on startup
if (process.env.NODE_ENV === 'development') {
  import('./src/lib/db/migrate').then(({ runMigrations }) => {
    runMigrations()
  })
}

export default nextConfig
```

**Note**: Next.js doesn't have a built-in "on startup" hook for arbitrary code. Alternative approaches:
- Use a custom server (`server.ts`) that runs migrations before starting Next.js
- Add a `predev` script in package.json: `"predev": "npm run db:migrate"`
- Use the `next.config.ts` approach above (runs at config load time)

The `predev` script approach is simplest and most reliable.

## Phase 6: Wire Database into Auth.js

### Overview

Update `auth.ts` to query the database for user authentication instead of using the `SEED_USERS` array.

### Changes Required:

#### 1. Update auth.ts to use database

**File**: `auth.ts`

**Intent**: Replace the `SEED_USERS` lookup with a database query via Drizzle.

**Contract**: Update the file:

```typescript
import NextAuth from "next-auth"
import CredentialsProvider from "next-auth/providers/credentials"
import bcrypt from "bcryptjs"
import { findUserByUsername } from "@/lib/db/schema"

export const { handlers, signIn, signOut, auth } = NextAuth({
  providers: [
    CredentialsProvider({
      name: "Credentials",
      credentials: {
        username: { label: "Username", type: "text" },
        password: { label: "Password", type: "password" }
      },
      async authorize(credentials) {
        if (!credentials?.username || !credentials?.password) {
          return null
        }

        const user = await findUserByUsername(credentials.username)
        if (!user) return null

        const validPassword = await bcrypt.compare(
          credentials.password as string,
          user.passwordHash
        )

        if (!validPassword) return null

        return { 
          id: user.id.toString(), 
          name: user.username, 
          email: `${user.username}@example.com` 
        }
      }
    })
  ],
  session: {
    strategy: "jwt",
  },
  pages: {
    signIn: "/login",
  },
  callbacks: {
    async jwt({ token, user }) {
      if (user) {
        token.id = user.id
      }
      return token
    },
    async session({ session, token }) {
      if (token) {
        session.user.id = token.id as string
      }
      return session
    }
  }
})
```

**Note**: The return type now uses `user.id.toString()` since SQLite integers are numbers in JS, but Auth.js expects string IDs.

## Performance Considerations

- SQLite is file-based — no network latency, queries are fast for small datasets
- Drizzle ORM adds minimal overhead (type-safe query building)
- better-sqlite3 is synchronous — blocks the event loop, but acceptable for low-traffic LAN app
- For future growth: connection pooling not needed (single-file DB), but could migrate to PostgreSQL if needed

## Migration Notes

- This plan transitions from `SEED_USERS` array to database-backed users
- The same bcrypt hashes are used for seed data — no password reset needed
- Auth.js session JWT continues to work — only the credential lookup changes
- If migration fails, revert `auth.ts` to use `SEED_USERS` temporarily

## Open Risks & Assumptions

- **better-sqlite3 native bindings**: May require compilation (node-gyp). If this fails, fallback to `@libsql/client` (pure JS SQLite).
- **Dev/Prod parity**: Using same SQLite file in dev and prod (if deployed). For production, consider PostgreSQL.
- **Backup strategy**: SQLite file must be backed up manually or via script — no automatic cloud backup.
