# Database Scaffold — Plan Brief

> Full plan: `context/changes/database-scaffold/plan.md`
> Research: N/A (no research doc provided)

## What & Why

Implement the database foundation (F-02) using Drizzle ORM + SQLite to replace the temporary seed-user approach from F-01. This provides persistent, database-backed user authentication while keeping the low-complexity goal for this LAN-only, 2-user deployment.

## Starting Point

The auth-scaffold (F-01) implemented Auth.js v5 with credentials provider, but uses an in-memory `SEED_USERS` array for credential lookup. The `src/lib/db/` directory exists with only a placeholder schema file. No database library is installed.

## Desired End State

- Drizzle ORM configured with SQLite (file-based, zero external services)
- `users` table with minimal schema: `id`, `username`, `password_hash`, `created_at`
- Auto-migrations run on dev startup via `predev` script
- `auth.ts` queries the database for user lookup (same bcrypt comparison pattern)
- Fail loudly on startup if database is missing or corrupt
- Same test users work (`admin`/`wedding2026`)

## Scope

**In scope:**
- Drizzle ORM + Drizzle Kit installation
- SQLite database connection singleton
- Users table schema definition
- Migration generation and auto-execution
- Auth.js integration (database-backed user lookup)
- Seed script for initial test users
- Fail-loud error handling on startup

**Out of scope:**
- Password recovery/reset flows
- Additional tables (guests, services, etc.)
- Database admin UI (use `sqlite3` CLI or Drizzle Studio)
- Unit tests for database queries
- Auth.js adapter pattern (JWT sessions stay in Auth.js)
- PostgreSQL or Docker setup

## Architecture / Approach

Drizzle ORM sits between the app and SQLite, providing type-safe queries. The database singleton (`src/lib/db/index.ts`) exports a `db` instance used throughout the app. Migrations are SQL files generated by `drizzle-kit` from the schema definition, run automatically before dev server starts. Auth.js queries the database via `findUserByUsername()` helper, then uses bcryptjs for password comparison (unchanged from F-01).

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌──────────┐
│  Auth.js    │────▶│  findUser    │────▶│  Drizzle    │────▶│  SQLite  │
│  (auth.ts)  │     │  (schema.ts) │     │  (db/index) │     │  (.data) │
└─────────────┘     └──────────────┘     └─────────────┘     └──────────┘
```

## Phases at a Glance

| Phase | What it delivers                    | Key risk                                |
| ----- | ----------------------------------- | --------------------------------------- |
| 1. Install Dependencies | drizzle-orm, better-sqlite3, drizzle-kit | Native bindings (node-gyp) may fail to compile |
| 2. Configure DB Connection | SQLite singleton in `src/lib/db/index.ts` | Directory creation must happen before DB open |
| 3. Define Users Schema | Drizzle table definition + type exports | Column naming (camelCase vs snake_case) consistency |
| 4. Drizzle Kit & Migrations | Config + initial migration SQL | Migration folder must exist |
| 5. Auto-Migration on Dev Start | `predev` script runs migrations | Must be idempotent (safe to run twice) |
| 6. Wire into Auth.js | Database query replaces SEED_USERS | Same bcrypt comparison, just different data source |
| 7. Error Handling | Fail loudly on corrupt/missing DB | Clear error messages guide the fix |

**Prerequisites:** F-01 (auth-scaffold) complete — Auth.js v5 configured and working
**Estimated effort:** ~2 sessions across 7 phases

## Open Risks & Assumptions

- **better-sqlite3 native compilation**: May require node-gyp. If it fails, fallback to `@libsql/client` (pure JS SQLite).
- **SQLite file backup**: Manual or scripted — no automatic cloud backup like managed PostgreSQL.
- **Single-file concurrency**: SQLite locks the file during writes; acceptable for 2 users, but not scalable to high concurrency.

## Success Criteria (Summary)

- `npm run dev` creates/connects to SQLite and runs migrations automatically
- `npm run db:seed` populates initial test users
- Login with `admin`/`wedding2026` works via database-backed auth
- Invalid credentials are rejected
- Corrupt/missing database fails loudly with clear error message
